Cimientos del proyecto

02
Inicialización de Hilos y Mutexes
Estructuras de datos y sincronización base
Estructuras
▼

Definir la estructura principal de datos: t_coder (id, compile_count, last_compile_time, estado) y t_dongle (mutex, cond, cooldown_until, wait_queue). Inicializar con malloc y memset.

Lanzar los hilos de cada coder con pthread_create. Crear un mutex por dongle y uno global para el log. Asegurarse de que todos los hilos arranquen y terminen correctamente con pthread_join. Recordar: hay tantos dongles como coders, dispuestos en círculo.

pthread_create pthread_join pthread_mutex_init malloc / free
Los N hilos se lanzan, imprimen su número de coder y terminan sin leaks ni crashes.
03
Rutina del Coder — El Ciclo de Vida
Compilar → Depurar → Refactorizar → repetir
Lógica Central
▼

Implementar la función de rutina del coder: intentar adquirir los dos dongles (izquierdo y derecho), imprimir has taken a dongle dos veces, luego simular la compilación con usleep, imprimir is compiling, soltar los dongles, pasar a debugging y refactoring.

Usar gettimeofday para los timestamps de los logs. Proteger toda salida a pantalla con el mutex de log para que las líneas no se mezclen. Registrar el last_compile_start de cada coder al inicio de cada compilación.

usleep gettimeofday pthread_mutex_lock/unlock write / printf
El ciclo compile→debug→refactor funciona con logs en el formato correcto y timestamps precisos en ms.
04
Hilo Monitor — Detección de Agotamiento
La muerte del programador en ≤ 10ms
Vigilante
▼

Crear un hilo monitor separado que en un bucle ajustado comprueba constantemente si now - last_compile_start ≥ time_to_burnout para cada coder. Al detectarlo, imprimir el log burned out y levantar una bandera global de parada para que todos los hilos terminen limpiamente.

También implementar la condición de parada exitosa: cuando todos los coders han compilado al menos number_of_compiles_required veces. El monitor vigila ambas condiciones. El log de burnout debe aparecer en menos de 10ms del momento real — esto requiere que el monitor use sleeps muy cortos (< 1ms).

pthread_create gettimeofday usleep ≤ 10ms de precisión
El programa detecta el agotamiento y lo imprime en ≤ 10ms. La simulación se detiene limpiamente en ambos casos de fin.
05
Cola de Prioridad — Implementar el Montículo
Montículo mínimo en C89 para FIFO/EDF
Estructuras de Datos
▼

El enunciado exige implementar una cola de prioridad con montículo para los planificadores. Escribir las funciones: heap_push(), heap_pop(), heap_peek(), y las auxiliares sift_up() / sift_down(). El montículo almacena nodos con el id del programador y su clave de prioridad.

Para FIFO, la clave es el timestamp de llegada a la cola. Para EDF, la clave es el plazo límite (last_compile_start + time_to_burnout). El mismo montículo sirve para ambos, solo cambia la función de comparación. Usar un montículo mínimo (menor clave = mayor prioridad).

malloc / free memset Montículo mínimo Patrón comparador
Insertar y extraer elementos del montículo devuelve siempre el de menor clave. Pruebas unitarias básicas pasan correctamente.
06
Planificadores FIFO/EDF y Enfriamiento del Dongle
Arbitración justa + período de enfriamiento
Planificación
▼

Integrar el montículo en la lógica de cada dongle. Cuando un programador quiere un dongle: se añade al montículo de espera con su prioridad, luego espera con pthread_cond_wait. Cuando el dongle se libera, el hilo que lo liberó extrae el de mayor prioridad del montículo y le hace pthread_cond_signal.

Implementar el enfriamiento del dongle: al liberar un dongle, registrar release_time. Si un programador intenta tomarlo antes de que pasen dongle_cooldown ms, espera con pthread_cond_timedwait hasta que el enfriamiento expire. El enfriamiento es parte del estado del dongle, no del programador.

pthread_cond_wait pthread_cond_timedwait pthread_cond_broadcast Orden FIFO Plazo límite EDF
Con FIFO los programadores acceden en orden de llegada. Con EDF, quien tiene el plazo más próximo va primero. El enfriamiento impide la reutilización inmediata.
07
Bloqueos Mutuos, Vivacidad y Pruebas de Estrés
Robustez y casos extremos
Robustez
▼

Probar el caso clásico de bloqueo mutuo del problema del filósofo: dos programadores adyacentes ambos agarran su dongle izquierdo y esperan el derecho eternamente. Solución: establecer un orden canónico de adquisición (siempre tomar primero el dongle de menor índice), rompiendo la circularidad de la espera (condición de Coffman).

Con EDF, verificar que no haya inanición: un programador con plazo lejano no puede esperar indefinidamente si hay dongle disponible. Probar con 1 programador, 2 programadores (caso con un solo dongle), y muchos. Usar Valgrind o AddressSanitizer para detectar carreras de datos y fugas de memoria.

Sin bloqueo mutuo Sin inanición Valgrind Analizador de hilos Casos límite
El programa es estable bajo cualquier combinación de parámetros. No se congela, no falla, no tiene carreras de datos detectables.
