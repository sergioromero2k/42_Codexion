Structures

Launch each coder's thread with pthread_create. Create one mutex per dongle plus a global log mutex. Ensure all threads start and finish cleanly with pthread_join. Remember: there are as many dongles as coders, arranged in a circle.

pthread_create pthread_join pthread_mutex_init malloc / free
All N threads launch, print their coder number, and exit cleanly — no leaks, no crashes.

04
Monitor Thread — Burnout Detection
Coder death detected in ≤ 10ms
Monitor
▼

Create a dedicated monitor thread that continuously checks whether now - last_compile_start ≥ time_to_burnout for each coder. On detection, print the burned out log and raise a global stop flag so all threads terminate cleanly.

Also implement the success stop condition: when every coder has compiled at least number_of_compiles_required times. The monitor watches both conditions. The burnout log must appear within 10ms of the actual event — this requires the monitor to use very short sleeps (< 1ms).

pthread_create gettimeofday usleep ≤ 10ms precision
The program detects burnout and prints it within ≤ 10ms. The simulation stops cleanly in both termination cases.
05
Priority Queue — Implementing the Heap
Min-Heap in C89 for FIFO/EDF
Data Structures
▼

The subject requires implementing a priority queue with a heap for the schedulers. Write the functions: heap_push(), heap_pop(), heap_peek(), and helpers sift_up() / sift_down(). The heap stores nodes containing the coder's id and its priority key.

For FIFO, the key is the arrival timestamp in the queue. For EDF, the key is the deadline (last_compile_start + time_to_burnout). The same heap serves both — only the comparator function changes. Use a Min-Heap (lower key = higher priority).

malloc / free memset Min-Heap Comparator pattern
Inserting and extracting from the heap always returns the lowest-key element. Basic unit tests pass correctly.
06
FIFO/EDF Schedulers & Dongle Cooldown
Fair arbitration + cooldown period
Scheduling
▼

Integrate the heap into each dongle's logic. When a coder wants a dongle: add them to the wait heap with their priority, then block with pthread_cond_wait. When the dongle is released, the releasing thread extracts the highest-priority waiter from the heap and wakes it with pthread_cond_signal.

Implement dongle cooldown: upon releasing a dongle, record release_time. If a coder tries to take it before dongle_cooldown ms have elapsed, they wait with pthread_cond_timedwait until the cooldown expires. The cooldown is part of the dongle's state, not the coder's.

pthread_cond_wait pthread_cond_timedwait pthread_cond_broadcast FIFO ordering EDF deadline
With FIFO, coders access in arrival order. With EDF, whoever has the nearest deadline goes first. Cooldown prevents immediate reuse.
07
Deadlocks, Liveness & Stress Testing
Robustness and edge cases
Robustness
▼

Test the classic deadlock scenario from the dining philosophers problem: two adjacent coders both grab their left dongle and wait forever for the right one. Solution: enforce a canonical acquisition order (always take the lower-index dongle first), breaking the circular wait (Coffman's condition).

With EDF, verify there is no starvation: a coder with a distant deadline cannot wait indefinitely if a dongle is available. Test with 1 coder, 2 coders (single-dongle edge case), and many coders. Use Valgrind or AddressSanitizer to catch data races and memory leaks.

No deadlock No starvation Valgrind ThreadSanitizer Edge cases
The program is stable under any parameter combination. It never freezes, never crashes, and has no detectable data races.
